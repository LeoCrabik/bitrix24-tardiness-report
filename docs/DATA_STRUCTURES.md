# Структуры данных — Универсальные списки Битрикс24

Все данные хранятся на стороне портала в двух универсальных списках типа `lists` (IBLOCK_TYPE_ID = `"lists"`).  
Оба списка создаются при установке и закрыты от доступа рядовых пользователей.

---

## Список 1: Записи об опозданиях

**Символьный код (CODE):** `TARDINESS_APP_RECORDS`  
**Название:** `Опоздания (приложение)`

### Поля

| Код поля | Тип B24 | Тип данных | Описание |
|---|---|---|---|
| `NAME` | `S` | string | Системное название записи (авто, напр. `2026-04-28_user_42`) |
| `PROPERTY_USER_ID` | `N` | int | ID сотрудника в Битрикс24 |
| `PROPERTY_DATE` | `S:Date` | date | Дата рабочего дня |
| `PROPERTY_ACTUAL_START` | `S:DateTime` | datetime | Фактическое время начала рабочего дня |
| `PROPERTY_PLAN_START` | `S:DateTime` | datetime | Плановое время начала рабочего дня |
| `PROPERTY_LATE_MINUTES` | `N` | int | Количество минут опоздания |
| `PROPERTY_REASON` | `S` | string | Причина опоздания (текст от сотрудника) |
| `PROPERTY_REASON_STATUS` | `L` | enum | Статус причины |
| `PROPERTY_MANAGER_ID` | `N` | int | ID руководителя, изменившего статус |
| `PROPERTY_RESOLVED_AT` | `S:DateTime` | datetime | Когда руководитель принял/отклонил |

### Значения `PROPERTY_REASON_STATUS`

| Код | Отображение |
|---|---|
| `PENDING` | Ожидает рассмотрения |
| `ACCEPTED` | Принято |
| `REJECTED` | Отклонено |
| `NONE` | Причина не указана (значение по умолчанию) |

### Индексирование / фильтрация
- Основные фильтры: `PROPERTY_USER_ID` + `PROPERTY_DATE` (диапазон)
- Уникальность записи: одна запись на пару (USER_ID, DATE)

## Список 2: Настройки приложения

**Символьный код (CODE):** `TARDINESS_APP_SETTINGS`  
**Название:** `Настройки (приложение опозданий)`

В списке хранится **одна запись** (singleton). Все настройки — в полях этой записи.

### Поля

| Код поля | Тип B24 | Тип данных | Описание |
|---|---|---|---|
| `NAME` | `S` | string | `"settings"` (константа) |
| `PROPERTY_TRACKED_USERS` | `S` | string | JSON-массив ID отслеживаемых сотрудников |
| `PROPERTY_MANAGERS` | `S` | string | JSON-массив ID руководителей |
| `PROPERTY_LATE_THRESHOLD` | `N` | int | Порог опоздания в минутах (по умолчанию 5) |
| `PROPERTY_SCHEDULE` | `S` | string | JSON-объект расписания по дням (см. ниже) |

### Формат `PROPERTY_SCHEDULE`

```json
{
  "1": { "enabled": true, "start": "09:00", "end": "18:00" },
  "2": { "enabled": true, "start": "09:00", "end": "18:00" },
  "3": { "enabled": true, "start": "09:00", "end": "18:00" },
  "4": { "enabled": true, "start": "09:00", "end": "18:00" },
  "5": { "enabled": true, "start": "09:00", "end": "18:00" },
  "6": { "enabled": false, "start": "09:00", "end": "18:00" },
  "7": { "enabled": false, "start": "09:00", "end": "18:00" }
}
```

Ключи: `1` = понедельник, `7` = воскресенье (ISO 8601).

---

## Права доступа к спискам

При создании обоих списков устанавливаются права:

| Группа | Права |
|---|---|
| Администраторы портала | Чтение, запись, удаление, администрирование |
| Все остальные | Нет доступа |

Приложение обращается к спискам через токен с правами администратора (application_token из установки).

# Функциональная спецификация: Отчёт по опозданиям

## 1. Описание продукта

Тиражное приложение для маркетплейса Битрикс24, которое:

- фиксирует опоздания сотрудников по данным модуля «Учёт рабочего времени» (timeman)
- предоставляет руководителю сводный отчёт с детализацией по каждому опозданию за любой период
- позволяет сотруднику указать причину опоздания
- позволяет руководителю принять или отклонить причину
- поддерживает экспорт в Excel по выбранным фильтрам
- хранит все данные на стороне портала (универсальные списки Битрикс24), не требует внешней БД

---

## 2. Роли пользователей

| Роль | Как определяется | Что видит |
|---|---|---|
| **Администратор** | `user.admin` возвращает `true` для токена пользователя | `SettingsPage` — настройка приложения |
| **Руководитель** | ID пользователя есть в поле `managers` настроек | `ReportPage` — отчёт + принятие/отклонение + экспорт |
| **Сотрудник** | Все остальные | `MyTardinessPage` — свои опоздания + ввод причины |

Один пользователь может совмещать роли (например, администратор + руководитель).

---

## 3. Настройки (экран администратора)

### 3.1 Отслеживаемые сотрудники
- Мультиселект пользователей портала
- Хранится: JSON-массив USER_ID в поле `TRACKED_USERS`

### 3.2 Руководители
- Мультиселект пользователей
- Руководитель получает доступ к полному отчёту и функциям принятия/отклонения
- Хранится: JSON-массив USER_ID в поле `MANAGERS`

### 3.3 Рабочий график
- Настройка «от» и «до» для каждого дня недели (Пн–Вс)
- Каждый день можно включить/выключить (суббота и воскресенье — выходные по умолчанию)
- По умолчанию: Пн–Пт, 09:00–18:00
- Используется для определения планового времени начала

### 3.4 Порог опоздания
- Количество минут, после которых сотрудник считается опоздавшим
- По умолчанию: 5 минут

## 4. Логика фиксации опоздания

### 4.1 Когда происходит проверка

Проверка запускается **по запросу** — без cron:
- Руководитель открывает отчёт (`GET /api/report`) → импорт данных за запрошенный период для всех отслеживаемых сотрудников
- Сотрудник открывает свои опоздания (`GET /api/my-tardiness`) → импорт данных за запрошенный период для себя

### 4.2 Источник данных

| Данные | Источник |
|---|---|
| Фактическое время начала за любой период | `timeman.timecontrol.reports.get(USER_ID, MONTH, YEAR)` → поле `days[].workday_date_start` |
| Плановое время начала | Расписание из настроек приложения (поле `SCHEDULE`, ключ = ISO день недели) |
| Проверка включён ли timeman | `timeman.settings(USER_ID)` → поле `UF_TIMEMAN` |

### 4.3 Алгоритм импорта за период

```
importTardinessForPeriod(userIds, dateFrom, dateTo):

  existingNames = getExistingRecordNames()  // один запрос, Set всех NAME в списке

  для каждого userId:
    tmSettings = timeman.settings(userId)
    если tmSettings.UF_TIMEMAN == false → пропустить

    для каждого месяца в [dateFrom..dateTo]:
      report = timeman.timecontrol.reports.get(USER_ID, MONTH, YEAR)  // пользовательский токен!
      
      для каждого дня из report.days[]:
        если нет workday_date_start → пропустить
        date = workday_date_start[0..9]   // "YYYY-MM-DD"
        если date вне [dateFrom..dateTo] → пропустить
        
        recordName = "${date}_user_${userId}"
        если recordName в existingNames → пропустить (уже записано)
        
        planStart = schedule[dayOfWeek]   // из настроек; если день выходной → пропустить
        tz = timezone из workday_date_start
        lateMinutes = (actualStart - planStart+tz) / 60 секунд
        
        если lateMinutes > lateThreshold:
          createRecord(userId, date, actualStart, planStart, lateMinutes)
          existingNames.add(recordName)
```

### 4.4 Принятие причины

Если руководитель принял причину — запись сохраняется со статусом `ACCEPTED`. При подсчёте «незакрытых» опозданий такие записи не учитываются.

## 5. Интерфейс руководителя — Отчёт

### 5.1 Фильтры

- **Период:** диапазон дат (по умолчанию — текущий месяц)
- **Сотрудник:** мультиселект из числа отслеживаемых сотрудников

### 5.2 Структура таблицы

Записи сгруппированы по сотрудникам. Заголовок группы содержит имя и количество незакрытых опозданий.

Строки детализации:
| Дата | Плановое | Фактическое | Опоздание | Причина | Статус | Действия |
|---|---|---|---|---|---|---|
| 15.04.2026 | 09:00 | 09:13 | 13 мин | — | Без причины | — |
| 03.03.2026 | 09:00 | 09:32 | 32 мин | «Пробки» | Ожидает | [Принять] [Отклонить] |

Кнопки [Принять] / [Отклонить] показываются только при статусе `PENDING`.

### 5.3 Экспорт в Excel

- Кнопка «Экспорт» применяет текущие фильтры
- Колонки: Сотрудник, Дата, Плановое время, Фактическое время, Опоздание (мин), Причина, Статус

---

## 6. Интерфейс сотрудника

- Таблица своих опозданий (только свои записи)
- Фильтр по периоду
- Для записей без причины — поле ввода и кнопка «Отправить»
- После отправки причины редактирование запрещено (статус меняется на `PENDING`)

## 7. Установка приложения

При установке (событие `ONAPPINSTALL`):
1. Сохранить токены доступа портала в Redis
2. Проверить существование списка `TARDINESS_APP_RECORDS` — если нет, создать со всеми полями
3. Проверить существование списка `TARDINESS_APP_SETTINGS` — если нет, создать + singleton с дефолтами
4. Сохранить ID обоих списков через `app.option.set`
5. Ответить HTML с `BX24.installFinish()` — обязательно, иначе Б24 считает установку незавершённой

При **переустановке** (списки уже существуют) — пересоздание пропускается, токены обновляются.

## 8. Требования к маркетплейсу

- Тип приложения: тиражное (массовое)
- Размещение: iframe в Битрикс24
- Scopes: `timeman`, `lists`, `user`, `department`
- Языки: русский
- Поддержка: Облако и Коробка Битрикс24

---

## 9. Ограничения и допущения

1. **Только рабочий график из настроек** — индивидуальные графики Б24 (`timeman.schedule.get`) не используются (запланировано на v2)
2. **Праздничные дни не учитываются** — производственный календарь в v2
3. **Нет пагинации** — все записи загружаются одним запросом; при большом объёме данных возможна задержка
4. **Пользователи без timeman пропускаются** — если `UF_TIMEMAN = false`
5. **Порог единый для всех** — индивидуальные пороги в v2

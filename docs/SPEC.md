# Функциональная спецификация: Отчёт по опозданиям

## 1. Описание продукта

Тиражное приложение для маркетплейса Битрикс24, которое:

- автоматически фиксирует опоздания сотрудников по данным модуля «Учёт рабочего времени» (timeman)
- предоставляет руководителю сводный отчёт с детализацией по каждому опозданию
- позволяет сотруднику указать причину опоздания
- позволяет руководителю принять или отклонить причину (если причина принята — опоздание не засчитывается)
- поддерживает экспорт в Excel по выбранным фильтрам
- хранит все данные на стороне портала (универсальные списки Битрикс24), не требует внешней БД

---

## 2. Роли пользователей

| Роль | Описание |
|---|---|
| **Администратор** | Портальный администратор. Настраивает приложение: выбирает отслеживаемых сотрудников, руководителей, график, порог опоздания. |
| **Руководитель** | Указывается администратором. Видит отчёт по опозданиям, принимает/отклоняет причины, экспортирует Excel. |
| **Сотрудник** | Видит свои опоздания, может ввести причину по каждому факту. |

---

## 3. Настройки (экран администратора)

### 3.1 Отслеживаемые сотрудники
- Мультиселект пользователей портала (через стандартный диалог BX24.selectUsers или кастомный)
- Можно добавить/удалить в любой момент
- Хранится: список USER_ID в списке настроек

### 3.2 Руководители
- Мультиселект пользователей
- Руководитель получает доступ к полному отчёту и функциям принятия/отклонения
- Хранится: список USER_ID в списке настроек

### 3.3 Рабочий график
- Настройка «от» и «до» для каждого дня недели (Пн–Вс)
- Каждый день можно включить/выключить независимо (например, суббота и воскресенье — выходные)
- По умолчанию: Пн–Пт, 09:00–18:00
- Примечание: при наличии графика в Битрикс24 (`timeman.schedule.get`) рекомендуется брать данные оттуда, а не дублировать; настройка в приложении служит как явный override для сотрудников, не привязанных к графику Б24

### 3.4 Порог опоздания
- Количество минут от начала рабочего дня, после которых сотрудник считается опоздавшим
- Значение по умолчанию: 5 минут
- Диапазон: 1–120 минут

### 3.5 Источник времени начала дня
- Данные берутся из модуля timeman Битрикс24 (фактическое время открытия рабочего дня сотрудником)
- Если сотрудник не открыл рабочий день до конца рабочего дня — фиксируется как отсутствие (не опоздание, не учитывается в данном отчёте; тема для будущей доработки)

## 4. Логика фиксации опоздания

### 4.1 Источник данных
Время начала рабочего дня берётся из учёта рабочего времени Битрикс24 (`timeman`). Приложение подписывается на события открытия рабочего дня и/или выполняет периодическую проверку через cron (см. [ARCHITECTURE.md](ARCHITECTURE.md)).

### 4.2 Алгоритм
```
для каждого рабочего дня D и каждого отслеживаемого сотрудника S:
  1. Определить плановое время начала рабочего дня T_plan (из настроек приложения или графика Б24)
  2. Определить фактическое время начала рабочего дня T_actual (из timeman)
  3. Если T_actual не задано (день не открыт) → пропустить
  4. delta = T_actual - T_plan (в минутах)
  5. Если delta > порог_опоздания → зафиксировать опоздание в список «Опоздания»
```

### 4.3 Пересчёт при принятии причины
Если руководитель принял причину опоздания — запись сохраняется в базе, но при подсчёте количества опозданий и в экспорте помечается как «Принято» и не засчитывается в счётчик.

## 5. Интерфейс руководителя — Отчёт

### 5.1 Фильтры

| Фильтр | Тип |
|---|---|
| Период | Диапазон дат или выбор целого календарного месяца (переключатель режима) |
| Сотрудник | Мультиселект из числа отслеживаемых сотрудников |

### 5.2 Структура таблицы

Таблица сгруппирована по сотрудникам. Для каждого сотрудника:

**Строка-заголовок группы** (colspan):
| Сотрудник | Кол-во опозданий |
|---|---|
| Иванов Иван | 3 |

*Кол-во опозданий — только незакрытые (без принятой причины)*

**Строки детализации** (по каждому факту опоздания):
| Дата опоздания | Время начала дня | На сколько минут опоздал | Причина | Действие |
|---|---|---|---|---|
| 28.04.2026 | 09:47 | 47 мин | «Пробки» | [Принять] [Отклонить] |
| 25.04.2026 | 09:12 | 12 мин | — | — |

**Правила отображения кнопок:**
- Нет причины от сотрудника → кнопки не показываются
- Есть причина, статус «Ожидает» → обе кнопки активны
- Статус «Принято» → кнопка «Принять» подсвечена (активна визуально), «Отклонить» — неактивна
- Статус «Отклонено» → кнопка «Отклонить» подсвечена, «Принять» — неактивна

### 5.3 Экспорт в Excel
- Кнопка «Экспорт в Excel» — применяет текущие фильтры
- Структура файла:
  - Строка-заголовок с периодом и датой формирования
  - Колонки: Сотрудник, Дата, Время начала дня, Опоздание (мин), Причина, Статус
  - Строки сгруппированы по сотруднику (объединение ячеек в колонке «Сотрудник»)

## 6. Интерфейс сотрудника

- Таблица своих опозданий (только свои записи)
- Фильтр по периоду/месяцу
- Колонки: Дата, Время начала дня, Опоздание (мин), Причина, Статус
- Для записей без причины — поле ввода текста и кнопка «Отправить»
- Для записей с уже заполненной причиной — текст только для чтения + статус
- Редактирование причины после отправки запрещено

## 7. Установка приложения

При установке (событие `ONAPPINSTALL`):
1. Проверить существование универсального списка «Опоздания» (по символьному коду `TARDINESS_APP_RECORDS`)
2. Если нет — создать список + все поля (см. [DATA_STRUCTURES.md](DATA_STRUCTURES.md))
3. Проверить существование списка «Настройки» (`TARDINESS_APP_SETTINGS`)
4. Если нет — создать список + поля + создать единственную запись с дефолтными настройками
5. Оба списка создаются с правами доступа только для администраторов портала (группа «Администраторы» → полный доступ, остальным — нет доступа)
6. Сохранить `application_token` из параметров установки

## 8. Требования к маркетплейсу

- Тип приложения: тиражное (массовое)

- Языки: русский (основной), English (вторичный, по возможности)
- Поддержка: Облако и Коробка Битрикс24

---

## 9. Ограничения и допущения

1. Приложение не создаёт/изменяет данные timeman — только читает
2. Если у сотрудника нет включённого учёта рабочего времени (`UF_TIMEMAN = false`) — он пропускается при проверке опозданий
3. Если сотрудник использует свободный график (`UF_TM_FREE = true`) — логика применяется, но на основе настроек приложения, а не его личного графика
4. Праздничные дни и производственный календарь в текущей версии не учитываются (планируется в v2)
5. Часовой пояс берётся из настроек портала

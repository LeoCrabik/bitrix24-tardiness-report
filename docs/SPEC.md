# Функциональная спецификация: Отчёт по опозданиям

## 1. Описание продукта

Тиражное приложение для маркетплейса Битрикс24, которое:

- фиксирует опоздания сотрудников по данным модуля «Учёт рабочего времени» (timeman)
- предоставляет руководителю сводный отчёт с детализацией по каждому опозданию
- позволяет сотруднику указать причину опоздания
- позволяет руководителю принять или отклонить причину (если причина принята — опоздание не засчитывается)
- поддерживает экспорт в Excel по выбранным фильтрам
- хранит все данные на стороне портала (универсальные списки Битрикс24), не требует внешней БД

---

## 2. Роли пользователей

| Роль | Описание |
|---|---|
| **Администратор** | Портальный администратор. Настраивает приложение: выбирает отслеживаемых сотрудников, руководителей, график, порог опоздания. |
| **Руководитель** | Указывается администратором. Видит отчёт по опозданиям, принимает/отклоняет причины, экспортирует Excel. |
| **Сотрудник** | Видит свои опоздания, может ввести причину по каждому факту. |

---

## 3. Настройки (экран администратора)

### 3.1 Отслеживаемые сотрудники
- Мультиселект пользователей портала
- Можно добавить/удалить в любой момент
- Хранится: JSON-массив USER_ID в списке настроек

### 3.2 Руководители
- Мультиселект пользователей
- Руководитель получает доступ к полному отчёту и функциям принятия/отклонения
- Хранится: JSON-массив USER_ID в списке настроек

### 3.3 Рабочий график
- Настройка «от» и «до» для каждого дня недели (Пн–Вс)
- Каждый день можно включить/выключить независимо (суббота и воскресенье — выходные по умолчанию)
- По умолчанию: Пн–Пт, 09:00–18:00
- Используется как фолбэк, если у сотрудника не настроен индивидуальный порог в timeman

### 3.4 Порог опоздания
- Количество минут от начала рабочего дня, после которых сотрудник считается опоздавшим
- Значение по умолчанию: 5 минут
- Диапазон: 1–120 минут

## 4. Логика фиксации опоздания

### 4.1 Когда происходит проверка

Проверка запускается **по запросу пользователя** — не через cron:
- Руководитель открывает отчёт (`GET /api/report`) → backend проверяет опоздания за сегодня для всех отслеживаемых сотрудников (если в фильтре есть сегодняшняя дата)
- Сотрудник открывает свои опоздания (`GET /api/my-tardiness`) → backend проверяет своё опоздание за сегодня

Если запись за сегодня уже существует — повторная проверка не выполняется.

### 4.2 Источник данных

| Данные | Источник |
|---|---|
| Плановое время начала | `timeman.settings → UF_TM_MAX_START` (индивидуальный порог пользователя), при отсутствии — расписание из настроек приложения |
| Фактическое время начала | `timeman.open → TIME_START` (единственный доступный метод REST API) |

**Важно:** REST API Битрикс24 не предоставляет read-only метода для получения времени начала рабочего дня. Метод `timeman.open`:
- если день уже открыт — возвращает `TIME_START` без изменений
- если день не открыт — открывает его; `TIME_START = текущее время` (сотрудник опоздал или отсутствует)

### 4.3 Алгоритм

```
для каждого отслеживаемого сотрудника S:
  1. Проверить: есть ли уже запись за сегодня? Если да → пропустить
  2. Получить UF_TIMEMAN из timeman.settings. Если false → пропустить (timeman выключен)
  3. Определить T_plan:
     а. UF_TM_MAX_START из timeman.settings (индивидуальный порог)
     б. Если не задан → время из расписания приложения (PROPERTY_SCHEDULE)
  4. Вызвать timeman.open(USER_ID=S) → получить T_actual (TIME_START)
  5. Если T_actual не за сегодня → пропустить (день не открыт)
  6. delta = T_actual - T_plan (в минутах)
  7. Если delta > PROPERTY_LATE_THRESHOLD → создать запись в TARDINESS_APP_RECORDS
```

### 4.4 Пересчёт при принятии причины

Если руководитель принял причину опоздания — запись сохраняется, но при подсчёте количества опозданий помечается как «Принято» и не засчитывается.

## 5. Интерфейс руководителя — Отчёт

### 5.1 Фильтры

| Фильтр | Тип |
|---|---|
| Период | Диапазон дат или выбор целого календарного месяца |
| Сотрудник | Мультиселект из числа отслеживаемых сотрудников |

### 5.2 Структура таблицы

Таблица сгруппирована по сотрудникам.

**Строка-заголовок группы:**
| Сотрудник | Кол-во опозданий |
|---|---|
| Иванов Иван | 3 |

*Кол-во опозданий — только незакрытые (без принятой причины)*

**Строки детализации:**
| Дата | Время начала | Опоздание | Причина | Действие |
|---|---|---|---|---|
| 28.04.2026 | 09:47 | 47 мин | «Пробки» | [Принять] [Отклонить] |
| 25.04.2026 | 09:12 | 12 мин | — | — |

**Правила отображения кнопок:**
- Нет причины → кнопки не показываются
- Есть причина, статус «Ожидает» → обе кнопки активны
- Статус «Принято» → «Принять» подсвечена, «Отклонить» неактивна
- Статус «Отклонено» → «Отклонить» подсвечена, «Принять» неактивна

### 5.3 Экспорт в Excel

- Кнопка «Экспорт в Excel» применяет текущие фильтры
- Колонки: Сотрудник, Дата, Время начала, Опоздание (мин), Причина, Статус
- Строки сгруппированы по сотруднику

## 6. Интерфейс сотрудника

- Таблица своих опозданий (только свои записи)
- Фильтр по периоду
- Колонки: Дата, Время начала, Опоздание (мин), Причина, Статус
- Для записей без причины — поле ввода и кнопка «Отправить»
- Редактирование причины после отправки запрещено

## 7. Установка приложения

При установке (событие `ONAPPINSTALL`):
1. Сохранить токены доступа портала в Redis
2. Проверить существование универсального списка «Опоздания» (код `TARDINESS_APP_RECORDS`)
3. Если нет — создать список + все поля (см. DATA_STRUCTURES.md)
4. Проверить существование списка «Настройки» (`TARDINESS_APP_SETTINGS`)
5. Если нет — создать список + поля + создать singleton-запись с дефолтными настройками
6. Сохранить ID созданных списков через `app.option.set`
7. Ответить `BX24.installFinish()` — сигнал Битрикс24 об успешной установке

## 9. Ограничения и допущения

1. Приложение не создаёт/изменяет данные timeman намеренно; `timeman.open` вызывается как единственный доступный способ получить время начала дня
2. Если у сотрудника выключен timeman (`UF_TIMEMAN = false`) — он пропускается
3. Если сотрудник не открыл рабочий день к моменту запроса руководителя — `timeman.open` откроет его; это считается очень поздним началом
4. Праздничные дни и производственный календарь в текущей версии не учитываются (v2)
5. Часовой пояс: `TIME_START` из `timeman.open` содержит смещение часового пояса пользователя

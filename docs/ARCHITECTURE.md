## Backend (Node.js / Express)

### Модули

| Модуль | Описание |
|---|---|
| `install.handler` | Обработка `ONAPPINSTALL`: сохранение токенов, создание списков |
| `oauth.service` | Обмен кода на токены, обновление access_token |
| `bitrix.client` | Обёртка над REST API Б24 (call с авторефрешем токена, callBatch) |
| `tardiness.service` | Бизнес-логика: проверка опозданий, CRUD записей и настроек |
| `export.service` | Генерация Excel (ExcelJS) |
| `api.routes` | HTTP-маршруты для фронтенда |

### API Routes

| Метод | URL | Описание |
|---|---|---|
| `GET` | `/api/me` | Роль текущего пользователя |
| `GET` | `/api/users` | Список пользователей для выбора в настройках |
| `GET` | `/api/settings` | Текущие настройки |
| `POST` | `/api/settings` | Сохранить настройки |
| `GET` | `/api/report` | Отчёт об опозданиях (с фильтрами) + on-demand проверка сегодня |
| `GET` | `/api/report/export` | Скачать Excel |
| `POST` | `/api/tardiness/:id/reason-status` | Принять/отклонить причину |
| `GET` | `/api/my-tardiness` | Мои опоздания + on-demand проверка сегодня |
| `POST` | `/api/my-tardiness/:id/reason` | Сотрудник указывает причину |

## Логика фиксации опозданий — on-demand подход

Нет фонового cron-задания. Проверка происходит при запросе данных:

```
Пользователь открывает страницу отчёта или своих опозданий
  ↓
Frontend: GET /api/report (или /api/my-tardiness)
  ↓
Backend — checkAndRecordTardiness(userId, today, settings):
  1. recordExists? → если да, пропустить
  2. timeman.settings(USER_ID) → UF_TIMEMAN, UF_TM_MAX_START
     → UF_TIMEMAN=false? → пропустить
  3. T_plan = UF_TM_MAX_START ?? расписание из PROPERTY_SCHEDULE
  4. timeman.open(USER_ID) → TIME_START
     → дата не сегодня? → пропустить (день не открыт)
  5. delta = TIME_START - T_plan (минуты)
  6. delta > lateThreshold? → создать запись в TARDINESS_APP_RECORDS
  ↓
Backend: вернуть записи из TARDINESS_APP_RECORDS
```

**Почему не cron:**
- REST API Б24 не имеет read-only метода для чтения времени начала рабочего дня
- `timeman.open` — единственный метод с `TIME_START`; если вызывать его из фона, он может открыть день сотрудника в произвольное время
- On-demand: проверка происходит только когда руководитель смотрит отчёт или сотрудник открывает свою страницу; к этому моменту день уже открыт или однозначно не открыт

## Frontend (React + TypeScript)

### Определение роли пользователя

При загрузке (`BX24.init()`):
1. `GET /api/me` → `{ userId, role: 'admin' | 'manager' | 'employee' }`
2. Показать соответствующую страницу

### Страницы

| Страница | Роль | Описание |
|---|---|---|
| `SettingsPage` | admin | Настройка отслеживаемых сотрудников, руководителей, расписания, порога |
| `ReportPage` | manager | Таблица опозданий с фильтрами, принятие/отклонение причин, экспорт |
| `MyTardinessPage` | employee | Свои опоздания, ввод причины |

### Открытие приложения в iframe

Битрикс24 открывает приложение POST-запросом на стартовый URL. Vite dev server не обрабатывает POST, поэтому proxy перехватывает такие запросы и делает `303 redirect` на GET.

### Технологии

| Библиотека | Назначение |
|---|---|
| `react` | UI |
| `typescript` | Типизация |
| `BX24.js` | SDK Битрикс24 (iframe auth, dialogs) |
| `@tanstack/react-query` | Кэш и запросы к backend |
| `antd` | UI-компоненты |
| `dayjs` | Работа с датами |
| `exceljs` | Генерация Excel (на backend) |

## Деплой (marketplace)

- Приложение размещается на внешнем хостинге (VPS/PaaS)
- Backend доступен по HTTPS
- URL обработчика установки и iframe регистрируются в карточке приложения
- Токены порталов хранятся в Redis на стороне сервера

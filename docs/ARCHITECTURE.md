## Backend (Node.js / Express)

### Модули

| Модуль | Описание |
|---|---|
| `install.handler` | Обработка `ONAPPINSTALL`: сохранение токенов, создание списков |
| `oauth.service` | Обмен кода на токены, обновление access_token |
| `bitrix.client` | Обёртка над REST API Б24 (методы call, callBatch) |
| `tardiness.service` | Бизнес-логика: расчёт опозданий, сохранение в список |
| `cron.service` | Периодическая проверка опозданий |
| `export.service` | Генерация Excel (ExcelJS) |
| `api.routes` | Маршруты для фронтенда |

### API Routes (внутренние, для фронтенда)

| Метод | URL | Описание |
|---|---|---|
| `GET` | `/api/settings` | Получить текущие настройки |
| `POST` | `/api/settings` | Сохранить настройки |
| `GET` | `/api/report` | Получить данные отчёта (с фильтрами) |
| `POST` | `/api/tardiness/:id/reason-status` | Принять/отклонить причину |
| `POST` | `/api/my-tardiness/:id/reason` | Сотрудник указывает причину |
| `GET` | `/api/report/export` | Скачать Excel |
| `GET` | `/api/users` | Список пользователей для выбора |

## Фиксация опозданий — детальный флоу

### Вариант A: Cron-job (основной для MVP)

```
Ежедневно в 13:00 (или configurable время):
  1. Для каждого портала (токен есть в store):
     а. Получить настройки из списка TARDINESS_APP_SETTINGS
     б. Для каждого отслеживаемого сотрудника:
        - Получить фактическое время начала сегодняшнего рабочего дня
          (через timeman API — исследовать метод при реализации)
        - Определить плановое время начала (из расписания настроек)
        - Проверить: delta > threshold?
        - Если да И запись за сегодня ещё не существует → создать запись в TARDINESS_APP_RECORDS
```

> **Примечание:** Точный метод получения фактического времени начала рабочего дня (check-in) для конкретного пользователя на конкретную дату необходимо уточнить при реализации. Кандидаты: `timeman.timecontrol.reports.get` (упоминается в документации как источник REPORT_ID), а также возможная недокументированная или внутренняя часть timeman API.

### Вариант B: Event-based (дополнение к cron)

```
При регистрации: подписаться на событие открытия рабочего дня (OnTimManUpdate или аналог)
При срабатывании события:
  - Извлечь user_id и timestamp открытия рабочего дня
  - Немедленно проверить: опоздание?
  - Если да → записать в TARDINESS_APP_RECORDS
```

## Frontend (React + TypeScript)

### Маршруты приложения

| Путь | Компонент | Роль |
|---|---|---|
| `/` | `App` | Определение роли, редирект |
| `/settings` | `SettingsPage` | Администратор |
| `/report` | `ReportPage` | Руководитель |
| `/my-tardiness` | `MyTardinessPage` | Сотрудник |

### Определение роли пользователя

При загрузке приложения (через `BX24.init()`):
1. Вызвать backend `/api/me` — возвращает текущего пользователя и его роль (admin / manager / employee)
2. Backend определяет роль по:
   - Является ли пользователь администратором портала (`user.get` → `ADMIN = Y`)
   - Находится ли USER_ID в списке менеджеров из настроек
   - Иначе — роль `employee`

| Библиотека | Назначение |

## Деплой (marketplace)

- Приложение размещается на внешнем хостинге (VPS/PaaS)
- Backend доступен по HTTPS
- URL обработчика установки и iframe регистрируются в карточке приложения на partner.1c-bitrix.ru
- Токены порталов хранятся в Redis (или PostgreSQL) на стороне сервера

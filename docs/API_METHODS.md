# Методы Битрикс24 REST API

Scope: `timeman`, `lists`, `user`, `department`

---

## Учёт рабочего времени (timeman)

### `timeman.settings`
Получить настройки рабочего времени конкретного пользователя.

**Параметры:**
- `USER_ID` — ID пользователя (необязательно, по умолчанию — текущий)

**Возвращает:**
| Поле | Тип | Описание |
|---|---|---|
| `UF_TIMEMAN` | bool | Включён ли учёт РВ для пользователя |
| `UF_TM_FREE` | bool | Свободный график |
| `UF_TM_MAX_START` | string | Максимальное время начала дня `HH:MM:SS` |
| `UF_TM_MIN_FINISH` | string | Минимальное время завершения `HH:MM:SS` |
| `UF_TM_MIN_DURATION` | string | Минимальная длительность рабочего дня |
| `ADMIN` | bool | Может ли управлять РВ других сотрудников |

**Использование в приложении:** Проверить, включён ли timeman для отслеживаемых сотрудников. Получить индивидуальный настроенный порог начала дня.

---

### `timeman.schedule.get`
Получить рабочий график по ID.

**Параметры:**
- `id` — ID графика (обязательно)

**Возвращает (ключевые поля):**
| Поле | Тип | Описание |
|---|---|---|
| `SCHEDULE_TYPE` | string | `FIXED` / `SHIFT` / `FLEXTIME` |
| `SHIFTS` | array | Массив смен |
| `SHIFTS[].WORK_TIME_START` | int | Начало смены в секундах от полуночи (напр. `28800` = 08:00) |
| `SHIFTS[].WORK_TIME_END` | int | Конец смены в секундах |
| `SHIFTS[].WORK_DAYS` | string | Рабочие дни: `"12345"` = Пн-Пт |
| `SCHEDULE_VIOLATION_RULES.MAX_EXACT_START` | int | Макс. начало в секундах (граница опоздания) |

**Использование в приложении:** Получить плановое время начала дня из графика Б24 (как основа или как источник для сравнения с настройками приложения).

---

### `timeman.timecontrol.settings.get`
Получить настройки модуля контроля времени.

**Возвращает:**
| Поле | Тип | Описание |
|---|---|---|
| `active` | bool | Модуль контроля времени включён |
| `minimum_idle_for_report` | int | Мин. время отсутствия для отчёта (мин) |
| `report_full_type` | string | Кто видит полный отчёт: `all` / `user` / `none` |

**Использование:** Проверить, активен ли модуль timeman на портале.

---

### `timeman.timecontrol.reports.settings.get`
Получить настройки отчётов + роль текущего пользователя.

**Возвращает:**
| Поле | Тип | Описание |
|---|---|---|
| `active` | bool | Модуль активен |
| `user_id` | int | ID текущего пользователя |
| `user_admin` | bool | Пользователь — администратор |
| `user_head` | bool | Пользователь — руководитель |
| `report_view_type` | string | `none` / `head` / `full` / `simple` |

**Использование:** Определить роль текущего пользователя при открытии приложения.

---

### `timeman.timecontrol.reports.users.get`
Получить список пользователей подразделения для отчётов.

**Параметры:**
- `DEPARTMENT_ID` — ID подразделения (для администратора/руководителя)

**Использование:** Получить список сотрудников в подразделении руководителя.

---

## ⚠️ Ограничение: получение фактического времени открытия рабочего дня

REST API Битрикс24 не предоставляет прямого метода для получения истории фактических времён открытия рабочего дня (`timeman.open`) для произвольного набора пользователей и периода.

**Решение — два подхода:**

**A. Event-based (предпочтительный):**
- Регистрация обработчика события на открытие рабочего дня
- Проверить наличие события `OnTimManUpdate` / аналогичного в Битрикс24 при реализации
- При срабатывании события — записать данные (user_id, timestamp) в список «Опоздания»

**B. Cron-based (резервный):**
- Раз в день (напр., в 13:00) проверять по всем отслеживаемым сотрудникам, открыт ли у них рабочий день
- Сравнивать фактическое время начала (если доступно через `timeman.get` или аналогичный метод) с плановым

## Пользователи

### `user.get`
Получить список пользователей с фильтрацией.

**Параметры:** `FILTER` (массив условий), `SELECT` (список полей), `start` (пагинация)

**Ключевые поля:**
| Поле | Описание |
|---|---|
| `ID` | ID пользователя |
| `NAME`, `LAST_NAME` | Имя и фамилия |
| `PERSONAL_PHOTO` | Фото |
| `WORK_POSITION` | Должность |
| `ACTIVE` | Активен ли пользователь |

**Использование:** Выбор отслеживаемых сотрудников и руководителей в настройках.

---

### `department.get`
Получить список подразделений.

**Возвращает:** `ID`, `NAME`, `PARENT` (ID родительского), `UF_HEAD` (ID руководителя)

**Использование:** Построение дерева подразделений для выбора сотрудников по отделам.

---

## Универсальные списки (lists)

### `lists.get`
Получить список/массив списков.

**Параметры:** `IBLOCK_TYPE_ID` = `"lists"`, `FILTER` (по коду/ID)

---

### `lists.add`
Создать новый универсальный список.

**Параметры:** `IBLOCK_TYPE_ID`, `FIELDS` (NAME, CODE, SORT и др.)

---

### `lists.field.add`
Создать поле универсального списка.

**Ключевые типы полей:**
| Тип | Код | Описание |
|---|---|---|
| Строка | `S` | Текст |
| Число | `N` | Число |
| Дата/Время | `S:DateTime` | Дата и время |
| Дата | `S:Date` | Только дата |
| Привязка к сотруднику | `S:employee` | Выбор пользователя |
| Список | `L` | Enum-значения |

---

### `lists.element.add`
Добавить элемент в список.

**Параметры:** `IBLOCK_TYPE_ID`, `IBLOCK_ID`, `ELEMENT_CODE`, `FIELDS`

---

### `lists.element.get`
Получить элементы списка с фильтрацией.

**Параметры:** `IBLOCK_TYPE_ID`, `IBLOCK_ID`, `FILTER`, `SELECT`

---

### `lists.element.update`
Обновить элемент списка (для изменения статуса/причины).

---

### `lists.element.delete`
Удалить элемент (не используется в основном флоу, но нужен для очистки при удалении настроек).

---

## Настройки приложения

### `app.option.get` / `app.option.set`
Хранение глобальных опций приложения (ключ-значение). Альтернатива списку настроек для простых параметров.

**Использование:** ID списков (`TARDINESS_APP_RECORDS`, `TARDINESS_APP_SETTINGS`) после их создания при установке.

---

## Установка приложения

### Событие `ONAPPINSTALL`
Вызывается сразу после установки приложения на портал.

**Важно:** В параметрах передаётся `application_token` — его необходимо сохранить для последующих вызовов от имени приложения.

**Алгоритм обработчика:**
1. Сохранить `application_token` и параметры портала (домен, токен доступа)
2. Создать универсальные списки (если не существуют)
3. Инициализировать запись настроек по умолчанию
